## Decisions and reasoning (data-scout)

| Decision | Why |
|----------|-----|
| `ioredis` as Redis client | TypeScript-friendly; clean stream (XADD) API; `enableOfflineQueue: false` for fail-loud emits |
| Named import `{ Redis }` from `ioredis` | Default import under `moduleResolution: NodeNext` resolves incorrectly for `new Redis()` — named import matches package exports |
| `node:crypto.randomUUID()` for `event_id` | Node 22+ native — no extra dependency for UUID generation |
| `schedule(0)` on first tick | First run has no cursor; immediately fetch all available events from MagicBall |
| `enableOfflineQueue: false` on Redis | Emit errors throw immediately, caught per-event; `emitted_at` stays null rather than silently queuing against a dead connection |
| Reset script included | Dev replay requires resetting Data Scout alongside MagicBall — dedup table would block re-ingestion of same `source_event_id` otherwise |
| Tags re-stringified on insert | MagicBall API returns `tags: string[]`; SQLite has no array type — `JSON.stringify` on write, parse on API read (same pattern as MagicBall) |

- Reset scripts per service are the right mental model; a **documented full reset order** (and/or a single orchestrated reset) would reduce ops mistakes — still open; `scripts/start.sh` now covers the first three services in order.

## Open issues — first / second / third order (concise)

**First order (single run / direct effects)**

- End-to-end for one `event_id`: ingest → IL pipeline (Gemini steps) → optional `events:notification_signal` **requires** MongoDB, Redis, valid `GEMINI_API_KEY`, and for a **full** path an available **Notification Gateway** at `NOTIFICATION_GW_BASE_URL` (relevance HTTP).
- `SKIP_INTELLIGENCE=1` in [scripts/start.sh](../../scripts/start.sh) prevents starting IL; it does **not** supply a mock gateway for runs you start with `pnpm dev:intelligence` alone.

**Second order (cross-service / ops)**

- **Notification Gateway** exists in `apps/notification-gateway`; full happy path requires seeded users + Redis + IL + Mongo + `GEMINI_API_KEY`.
- **Aggregated admin logs** work across services with normalized `GET /logs` shapes; IL still does not embed non-`ai_logs` pipeline failures in `/logs`.
- **IL `GET /logs`** remains AI-call-centric; use **`GET /pipeline/:event_id`** for step-level failure inspection.

**Third order (long-term / discipline / product rules)**

- **Lineage + replay + future A/B** depend on consistent `event_id`, `pipeline_version`, and coordinated resets across MagicBall SQLite, Data Scout SQLite, and IL Mongo — operator discipline and clear runbooks.
- **Do not** add silent fallbacks or masked retries in the pipeline without an **explicit** product decision and doc update — that would conflict with stated fail-fast rules.
- **Config discipline:** local vs cloud should differ by **env only** (URIs, secrets, `GEMINI_MANIFEST_PATH` as absolute in prod); [apps/intelligence-layer/.env.example](../../apps/intelligence-layer/.env.example) is the template; `start.sh` is not a production entrypoint.

**Why this direction**

| Decision | Why |

| New app `apps/notification-gateway` mirroring MagicBall/Data Scout | Single Fastify + SQLite + Redis pattern; avoids a fifth architectural style. |
